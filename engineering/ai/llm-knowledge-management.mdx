# LLM 驱动的个人知识管理

## 背景：为什么传统 RAG 不够

大多数人用 LLM 处理文档的方式是 RAG（检索增强生成）：上传文件，提问时检索相关片段，再生成答案。NotebookLM、ChatGPT 文件上传等产品都是这个逻辑。

RAG 的根本缺陷在于：**模型每次回答都从零开始"重新发现"知识，没有任何积累。** 如果问题需要综合多篇文档的内容，模型每次都得重新检索、重新拼接。

## Karpathy 的 Wiki 编译模式

Andrej Karpathy 提出了一种截然不同的思路：不在提问时检索原始文档，而是让 LLM **持续构建并维护一个永久性的 Wiki**——一套结构化、相互链接的 Markdown 文件。

### 核心理念

每当加入一份新资料，LLM 不只是编入索引，而是：
- 真正读懂内容、提取关键信息
- 将其融入已有的 Wiki 体系
- 更新实体页面、修订主题摘要
- 标注新旧内容的矛盾之处

**知识只需编译一次，之后持续保持更新**，而不是每次提问都重新推导。

> 打个比方：你每次运行程序是跑编译后的二进制，不是每次从源代码重新解释执行。知识管理也一样——编译一次，后面反复查就行了。

### 三层架构

| 层级 | 内容 | 职责 |
|------|------|------|
| **原始资料层** | 文章、论文、图片、数据文件 | 只读，LLM 不修改。事实来源 |
| **Wiki 层** | LLM 生成的 Markdown 文件（摘要、实体页面、概念页面、对比分析等） | 完全由 LLM 写作和维护 |
| **Schema 层** | 配置文件（如 `CLAUDE.md`、`AGENTS.md`），定义 Wiki 结构和工作流规范 | 你和 LLM 共同迭代完善 |

Schema 层是整个系统的关键配置——它告诉 LLM Wiki 的结构约定、录入/查询/检查时应遵循的工作流程。正是它让 LLM 从通用聊天机器人变成有纪律的 Wiki 维护者。Karpathy 强调这份文档应在实践中由人和 LLM 共同迭代，逐步摸索出最适合具体领域的方式。

### 三种核心操作

**录入（Ingest）**：将新资料放入原始资料库，LLM 阅读后写摘要页面、更新索引、更新相关实体和概念页面。一份资料可能涉及 10-15 个 Wiki 页面的更新。

**查询（Query）**：向 Wiki 提问，LLM 搜索相关页面后综合作答。关键洞见：**好的回答可以作为新页面写回 Wiki**，让探索过程也在知识库中不断积累。

**检查（Lint）**：定期让 LLM 做健康检查——排查矛盾、过时内容、孤立页面、缺失的交叉引用，以及可以通过搜索填补的信息空白。

### 索引与日志

- **index.md**：内容导向的目录——每个页面附链接和一句话摘要，按类别（实体、概念、来源等）组织，可选元数据（日期、来源数量）。LLM 每次录入时更新，回答问题时先读索引定位再深入阅读。中等规模（~100 份资料、数百页面）下效果出奇地好，无需搭建向量嵌入的 RAG 基础设施
- **log.md**：时间导向的操作记录，只追加不修改，记录每次录入、查询、检查。实用技巧：每条记录以固定前缀开头（如 `## [2026-04-02] ingest | 文章标题`），日志就变得可用 Unix 工具处理——`grep "^## \[" log.md | tail -5` 即可列出最近 5 条记录

## 实战：Claude Code + Obsidian 工作流

陆徐洲（公众号"硅基鹿鸣"）基于 Karpathy 的思路，用 Claude Code 对 43 篇公众号文章做了一次完整的知识编译实践。

### 具体步骤

1. **扫描生成清单**：Claude Code 扫描整个目录，列出所有文章的标题、字数、涉及的工具
2. **LLM 编译**：新建 `wiki/` 文件夹作为 Obsidian vault，Claude Code 逐篇阅读，每篇生成摘要卡片（3-5 句话），用 `[[双向链接]]` 标注工具名
3. **概念提取**：从 43 张卡片中提取高频概念，每个概念生成独立页面。例如 `[[Claude Code]]` 概念页自动列出 28 篇相关文章
4. **索引生成**：自动生成全局索引、按栏目/工具/时间线分类的索引
5. **断链修复**：扫描所有双向链接，查找无对应文件的断链，自动补建高频概念页

### 产出

73 个 .md 文件：43 张文章卡片 + 25 个概念页 + 5 张索引。摘要从原始 30 万字浓缩到 5 万字，概念关系物化成文件结构。

### 最有价值的环节：对知识库提问

- **覆盖矩阵**：发现 Hooks 系统、IDE 插件、CI/CD 集成、Memory 持久记忆完全零覆盖
- **关联发现**：OpenCode 有 9 篇、Superpowers 有 4 篇，但两者零交叉；MCP 出现 6 次却从没当过文章主角
- **选题推荐**：LLM 给出了 Claude Code Hooks 实战、MCP Server 横评、OpenCode + Superpowers 桥接等选题建议

## 为什么这套方法有效

维护知识库最繁琐的不是阅读和思考，而是**记账**——更新交叉引用、保持摘要时效、标注矛盾、维护一致性。人们放弃维护 Wiki，正是因为维护成本增速超过价值。

LLM 不会感到无聊，不会忘记更新交叉引用，可以一次操作同时修改 15 个文件。**人负责筛选资料、指引方向、提出好问题；LLM 负责其他一切。**

这与 Vannevar Bush 1945 年提出的 Memex（记忆延伸机器）一脉相承——私人的、主动维护的知识库，文档间的联系与文档本身同等重要。Bush 唯一没解决的问题是谁来负责维护，LLM 解决了这个问题。

## 适用场景

- **深度研究**：围绕课题持续钻研，增量构建有完整论点演进的综合性 Wiki
- **读书笔记**：每读完一章就归档，为人物、主题、情节建页面并梳理联系
- **个人成长**：追踪目标、健康、心理状态，从日记和笔记中建立结构化图景
- **企业/团队**：LLM 维护的内部 Wiki，输入来自 Slack、会议记录、项目文档
- **内容创作**：如上述实践，梳理已有作品、发现空白、规划新选题

## 实用工具

- **Obsidian**：Markdown 编辑器，Graph View 可视化知识网络，双向链接是核心。图谱视图是查看 Wiki 全貌的最佳方式——哪些页面是枢纽，哪些是孤岛
- **Obsidian Web Clipper**：浏览器插件，将网页转为 Markdown 存入原始资料库。配合 Obsidian 设置中的"下载当前文件的附件"功能，可一键将文章图片下载到本地，让 LLM 能直接查看和引用
- **Dataview**：Obsidian 插件，可对页面 YAML 前置元数据运行查询，生成动态表格和列表。如果 LLM 为页面添加了标签、日期、来源数量等元数据，Dataview 可以做结构化检索
- **Marp**：基于 Markdown 的幻灯片格式，Obsidian 有对应插件，可直接从 Wiki 内容生成演示文稿
- **Claude Code / Codex / OpenCode**：作为 LLM 执行引擎，负责编译、查询、维护 Wiki
- **qmd**：本地 Markdown 搜索引擎，支持 BM25/向量混合搜索和 LLM 重排序，完全本地运行。同时提供 CLI 接口和 MCP 服务。Wiki 规模扩大后可替代索引文件
- **Git**：Wiki 本质上就是一个 Git 仓库，全是 Markdown 文件。版本历史、分支管理、多人协作，全部开箱即用
